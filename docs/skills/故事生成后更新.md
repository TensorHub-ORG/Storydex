# 故事生成后更新

> 模板版本：2｜适用范围：任意 Storydex 小说项目｜正文变化后增量同步项目资产。

## 1. 技能用途

在正文新增或修改后，按依赖顺序同步角色、变量、物品、事实、关系、WIKI 和必要索引。

## 2. 适用触发

- Agent 完成续写、故事生成或用户保存重要剧情修改。
- 用户要求把最新剧情同步到项目资产。

## 3. 输入检查

- 本次新增或修改的正文范围和生成前状态。
- 相关角色、变量、WIKI、世界书和未解决冲突。

## 4. 证据规则

- 只处理本次正文实际产生的变化。
- 候选推断不能写入长期记忆或正式 WIKI。
- 角色、变量与 WIKI 的更新必须保持同一事实版本。

## 5. 允许落点

- `.storydex/characters/`
- `.storydex/memory/`
- `.storydex/wiki/`
- `.storydex/worldbook/`，仅新增正文明确建立的设定

## 6. 执行步骤

1. 提取正文增量中的明确事实、状态变化和候选推断。
2. 先执行变量思考，确定时间、地点、角色、关系、物品和事件变化。
3. 在同一次正文增量中直接同步有证据且可安全合并的角色、变量、物品、事实和关系；重大变化进入待确认。
4. 在角色和变量更新后整理 WIKI/知识图谱。
5. 输出变更摘要、证据、路径和未处理事项，确认正文未被二次改写。

## 7. 输出模板

```markdown
# 故事生成后更新报告
## 本次正文范围
## 变量变化
## 角色更新
## 物品、地点与事件
## WIKI / 知识图谱更新
## 已写入文件
- 路径 / 修改摘要 / 证据
## 待确认或未处理
## 连续性风险
```

## 8. 完成前自检

- [ ] 更新范围只覆盖本次正文变化。
- [ ] 角色、变量和 WIKI 更新顺序正确。
- [ ] 没有无证据的长期记忆写入。
- [ ] 正文没有被更新流程再次覆盖。

## 9. 安全边界

- 自动更新耗时较高时应分批处理。
- 用户显式要求延期时不应用记忆增量；需复核的删除、冲突和重大关系变化始终保留待确认。
- 重大关系、身份、生死和删除操作先确认。
- 不把中间分析和工具日志写入长期记忆。
