# Storydex 使用指南

本文是 Storydex 的基础使用说明，适合第一次使用 Storydex 的用户阅读，也适合作为 Storydex 内置 LLM 回答“怎么使用 Storydex”类问题时的事实来源。

## 1. Storydex 是什么

Storydex 是一个本地优先的小说工程写作系统。它不是单纯的聊天窗口，也不是普通文本编辑器，而是把小说项目当成一组可管理的创作资产：

- 正文章节和剧情片段
- 角色文件
- 变量与变量思考
- WIKI 与知识图谱
- 创作预设
- 版本控制
- Agent 执行记录

你可以把 Storydex 理解成：

```text
本地小说项目 + AI 协作写作 + 文件化资产管理 + 本地 Git 版本溯源
```

Storydex 的 Agent 基座是 Coomi。Coomi 负责理解用户请求、调用工具、流式执行和处理失败；Storydex 负责小说项目的目录、章节、角色、变量、WIKI、预设、版本记录等领域编排。

## 2. 桌面版运行环境

Storydex 桌面版采用：

```text
Electron 桌面壳 + 内置 Python 环境 + 内置 MinGit
```

普通用户不需要额外安装 Python，也不需要额外安装 Git。

### 内置 Python

桌面版会随程序携带后端运行所需的 Python 环境。启动 Storydex 时，桌面程序会自动启动本地后端服务。

### 内置 MinGit

桌面版会随程序携带 MinGit。Storydex 使用它为小说项目创建版本控制。

这不会修改用户系统的 Git 配置，也不会污染系统 PATH。Storydex 只会在自己的后端运行环境中优先使用内置 Git。

### 网络需求

Storydex 的项目文件、版本记录和大部分管理功能都在本地运行。需要联网的部分主要是：

- 调用 LLM 模型服务
- 首次配置或更新某些在线模型供应商信息
- 开发环境下安装或更新依赖

正式桌面版普通使用时，不应该要求用户自己运行 `pip install`。

## 3. 第一次启动建议流程

第一次打开 Storydex 时，建议按下面顺序完成初始化：

1. 打开 Storydex。
2. 选择一个小说项目目录，或者创建一个新的空目录。
3. 初始化 Storydex 项目骨架。
4. 配置 LLM Provider、API Key 和模型。
5. 选择默认章节目录模板。
6. 设置剧情片段数量和片段字数。
7. 决定是否开启自动更新变量、自动更新 WIKI。
8. 让 Agent 生成一小段测试剧情，确认写入、Diff 和版本控制正常。

推荐第一次测试时使用简单请求：

```text
请帮我创建一个新故事开头，生成 1 条剧情片段，约 800 字，并创建出现的新角色文件。
```

如果你还不熟悉 Storydex，建议先保持：

```text
片段数量：1
片段字数：800 到 2000
自动更新变量：关闭
自动更新 WIKI：关闭
```

自动更新变量需要较多耗时，建议每次仅生成单条剧情片段。

该开关控制非正文生成场景的默认行为。Agent 生成正文时，会在同一次增量写入中直接整理有正文证据且可安全合并的变量、事实、物品和关系；用户显式要求延期或需要复核的重大变更仍不会自动写入。

## 4. 小说项目目录

Storydex 的小说项目是一个普通本地文件夹。项目文件可以人工打开、编辑、备份，也可以由 Agent 自动管理。

一个典型项目会包含：

```text
chapters/              正文章节或剧情片段
characters/            角色文件
world/                 世界观、设定、地点、势力等
variables/             变量、变量思考和状态记录
wiki/ 或 .storydex/wiki/ WIKI 与知识图谱数据
presets/               创作预设
.storydex/             Storydex 项目配置、索引、模板和内部数据
.git/                  本地 Git 版本记录
```

实际目录可能因模板和项目设置不同而变化。基础骨架由 Storydex 维护，用户可以扩展目录，但不要手动删除 `.storydex/` 和 `.git/`。

## 5. Agent 能做什么

Storydex Agent 可以协助完成这些任务：

- 设计角色
- 更新角色文件
- 设计世界书或设定条目
- 设计剧本和剧情走向
- 续写或生成故事片段
- 整理变量思考
- 更新事实、关系和物品记录
- 整理 WIKI
- 更新知识图谱
- 整理项目目录
- 检查本轮修改
- 创建版本控制

适合的提示词例子：

```text
请根据当前章节继续写 1 条剧情片段，约 2000 字，保持当前文风。
```

```text
请整理最近三章中主角和反派关系的变化，并更新角色文件。
```

```text
请根据当前项目文件整理 WIKI，优先更新人物、地点和势力条目。
```

```text
请检查当前项目目录，把明显放错位置的角色设定和世界观设定整理到合适目录。
```

不推荐的提示词：

```text
随便写点。
```

```text
把所有东西都整理一下。
```

更好的方式是给 Agent 明确任务范围、目标文件、生成数量和是否允许更新变量或 WIKI。

## 6. 剧情生成流程

当你请求“生成剧情”“续写故事”“写章节片段”时，Storydex 会尽量按下面流程工作：

1. 判断这是故事创作或续写意图。
2. 检查当前项目章节结构。
3. 如果是全新故事，要求选择章节目录模板。
4. 读取 active 或 compiled safe 创作预设。
5. 读取近期章节、活跃角色、变量、事实和关系。
6. 按用户设置生成剧情片段。
7. 分析角色、物品、变量、事实、关系和 WIKI 的增量变化。
8. 为新出现的角色创建角色文件。
9. 缺失信息写为“未知”。
10. 直接应用本次正文中有证据且可安全合并的变量、事实等增量；需复核项保留待确认，再根据设置或用户确认更新 WIKI。
11. 写入项目文件。
12. 本轮执行结束后检查修改，并创建本地 Git 版本记录。

### 片段数量和片段字数

剧情生成支持设置：

```text
片段数量
片段字数
章节模板
```

建议：

- 想稳定推进剧情时，片段数量设为 1。
- 想批量生成草稿时，可以增加片段数量。
- 开启自动变量或自动 WIKI 时，建议片段数量保持 1。

## 7. 角色文件

剧情中出现的新角色应该自动创建角色文件。即使暂时没有足够信息，也应该创建基础文件，并把缺失字段写为“未知”。

角色文件会随剧情发展持续更新。Storydex 在生成上下文时通常只注入近期活跃角色，不会一次性把所有角色塞进 prompt。

如果你想让 Agent 更新角色，可以这样说：

```text
请根据最近两章，更新林澈和许清昼的角色文件，重点整理动机、关系变化和已知秘密。
```

## 8. 变量与变量思考

变量用于记录故事状态，例如：

- 人物关系
- 阵营立场
- 物品归属
- 伤势和状态
- 已知秘密
- 剧情旗标
- 世界规则变化

Storydex 的变量思考应优先使用可读、可维护的自然语言或 Markdown 形式。不要强行把变量思考写成难维护的固定 JSON 路径和值。

合适的变量思考应该说明：

- 发生了什么变化
- 哪些变化已经确定
- 哪些变化仍不确定
- 是否和旧设定冲突
- 是否建议用户确认后再写入

如果你希望 Agent 只思考变量、不直接更新，可以这样说：

```text
请只做变量思考，不要直接改变量文件。列出本章可能影响的角色关系、物品归属和剧情旗标。
```

## 9. WIKI 与知识图谱

WIKI 是故事事实的可视化和检索层。它不应该完全替代角色文件、变量文件和事实记忆。

WIKI 适合记录：

- 人物条目
- 地点条目
- 势力条目
- 重要物品
- 关键事件
- 世界规则
- 人物关系

知识图谱适合展示人物、地点、势力、物品和事件之间的关系。

如果你想整理 WIKI，可以这样说：

```text
请根据当前章节和角色文件整理 WIKI，优先更新人物、地点、势力和关键事件。
```

## 10. 创作预设

创作预设用于约束文风、剧情节奏、对白、人物行动、禁用内容和写作偏好。

Storydex 中的预设通常分为：

- library：预设库
- imported：导入预设
- blocked：禁用预设
- active：当前启用预设
- compiled safe preset：编译后的安全预设

只有 active 或 compiled safe preset 可以进入生成上下文。

如果你觉得生成内容风格不稳定，优先检查：

1. 当前启用的预设是否正确。
2. 是否启用了互相冲突的预设。
3. 预设内容是否过长或含糊。
4. 当前任务是否明确要求遵守某个风格。

## 11. 版本控制

Storydex 会使用本地 Git 为小说项目记录版本。

重要说明：

- Git 是纯版本控制。
- Storydex 不会自动推送到远程仓库。
- 桌面版会内置 MinGit，用户不需要安装 Git。
- 每次 Agent 执行结束时，如果检测到小说项目文件被修改，应自动创建本地提交。
- 如果没有文件变化，则不会创建无意义提交。

版本记录的作用：

- 查看本轮修改了哪些文件
- 查看 Diff
- 追溯某次 Agent 写入
- 在必要时恢复旧版本
- 理解故事项目的变化历史

如果你看到“本轮没有修改”，但确定 Agent 写了文件，可以检查：

1. 当前打开的是否是正确小说项目。
2. 文件是否写到了项目目录之外。
3. 后端是否正在使用内置 Git 或可用 Git。
4. 本轮执行是否尚未结束。

## 12. 修改预览和 Diff

执行过程中，Storydex 会显示本轮已修改内容。点击已修改内容，可以在中间文件区域查看 Diff。

Diff 只展示修改区域，不展示完整文件全文。这样更适合快速确认 Agent 改了什么。

如果当前没有修改文件，浮条会显示“暂无本轮修改”。

如果当前没有真实任务清单，浮条会显示“暂未规划”。Storydex 不应该为了显示任务而编造无意义任务。

## 13. 常见工作流

### 创建新故事

```text
请创建一个新故事开头，选择适合长篇连载的章节目录结构，生成 1 条约 2000 字的剧情片段。
```

### 续写当前章节

```text
请根据当前章节继续写 1 条剧情片段，约 2000 字。保持当前文风，注意角色动机和上一段结尾。
```

### 设计角色

```text
请设计一个新角色，作为主角早期的敌人。需要包含姓名、身份、外貌、动机、秘密和与主角的初始关系。
```

### 更新角色

```text
请根据最近两章更新所有活跃角色文件，重点整理关系变化、已知秘密和心理状态。
```

### 变量思考

```text
请对最近生成的剧情做变量思考，用自然语言列出人物关系、物品归属、剧情旗标和潜在冲突。不要输出固定 JSON。
```

### 更新 WIKI

```text
请根据当前项目文件更新 WIKI 和知识图谱，优先处理人物、地点、势力和关键事件。
```

### 整理目录

```text
请检查项目目录结构，把明显放错位置的设定、角色和章节文件整理到合适目录，不要破坏基础骨架。
```

## 14. 常见问题

### 我需要安装 Python 吗？

正式桌面版普通使用不需要。Storydex 会内置 Python 环境。

### 我需要安装 Git 吗？

正式桌面版普通使用不需要。Storydex 会内置 MinGit，并用于版本控制。

### Storydex 会把我的小说上传到远程仓库吗？

不会。Storydex 的 Git 版本记录是纯本地的。除非用户以后明确配置远程同步功能，否则不会自动推送远程。

### 为什么建议一次只生成一条剧情片段？

因为生成正文后还需要分析角色、变量、事实、关系和 WIKI 增量。片段越多，后续一致性处理越复杂。自动更新变量尤其耗时，建议每次仅生成单条剧情片段。

### 为什么新角色信息很多是“未知”？

这是正常的。剧情中出现了新角色，但暂时没有足够信息时，Storydex 仍应创建角色文件，并把缺失字段写为“未知”。后续剧情发展后再持续更新。

### 为什么任务清单显示“暂未规划”？

这表示本轮还没有收到真实任务规划事件。Storydex 不应该在前端编造任务。只有后端或 LLM 真正规划出的任务才会显示在任务清单里。

### 为什么 Diff 没有显示？

可能原因：

1. 本轮确实没有修改项目文件。
2. Agent 还在执行，修改账本尚未汇总。
3. 文件写入路径不在当前小说项目目录内。
4. Git 尚未初始化或不可用。
5. 后端没有收到写入工具的结构化修改路径。

如果是桌面封装版，应优先检查内置 MinGit 是否被正确加入后端运行环境。

### 我可以手动修改文件吗？

可以。Storydex 的项目文件是普通本地文件。手动修改后，建议在 Storydex 中刷新项目状态，并根据需要创建版本控制。

## 15. 给 Storydex LLM 的回答规则

当用户询问 Storydex 使用方法时，LLM 应优先检索并依据本文回答。

回答时应遵守：

1. 不要假设用户懂 Python、Git、FastAPI、Electron 或项目内部实现。
2. 先给可执行步骤，再补充原因。
3. 如果涉及版本记录，明确说明这是小说项目的本地 Git，不是 Storydex 源码仓库。
4. 如果涉及变量思考，优先推荐自然语言或 Markdown，不强制 JSON 路径和值。
5. 如果涉及剧情生成，提醒用户设置片段数量、片段字数和章节模板。
6. 如果涉及自动变量或自动 WIKI，提醒用户耗时会增加，建议单条剧情片段。
7. 如果用户问“为什么没有任务清单”，说明只有真实规划任务才显示，没有任务时显示“暂未规划”。
8. 如果用户问“是否需要安装 Python/Git”，说明正式桌面版会内置 Python 环境和 MinGit。
