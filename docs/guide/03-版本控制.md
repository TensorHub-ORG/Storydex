# 版本控制详细说明

`版本控制` 是 Storydex 为用户的小说项目提供的本地 Git 版本管理界面。它用于记录项目文件每一次重要变化，方便查看 Diff、追踪 Agent 写入、回退旧版本。

重点：这里管理的是小说项目的本地 Git 仓库，不是 Storydex 源码仓库，也不会自动推送到远程仓库。

## 入口

1. 打开一个 Storydex 小说项目。
2. 点击左侧活动栏的版本控制图标。
3. 打开 `版本控制` 面板。

如果当前没有打开项目，面板会提示先打开 Storydex 项目。

## 界面区域

| 区域 | 说明 |
| --- | --- |
| 顶部标题 `版本控制` | 当前模块名称。 |
| 项目名称 | 当前正在管理的小说项目。 |
| `提交仓库` | 当前本地 Git 仓库状态、分支、提交消息和提交按钮。 |
| `更改` | 当前未提交的文件变化列表。 |
| `图表` | 最近的本地提交历史。 |

## 顶部按钮

| 按钮/图标 | 作用 |
| --- | --- |
| `刷新` | 重新读取 Git 状态、变更文件和提交历史。 |

## 未初始化状态

如果项目还没有 Git 仓库，会看到 `初始化本地仓库`。

| 按钮 | 作用 |
| --- | --- |
| `初始化本地仓库` | 在当前小说项目中创建本地 `.git/` 仓库。初始化后，Storydex 才能记录版本、展示 Diff 和提交图表。 |

初始化后，Storydex 会使用小说项目内的本地 Git 记录变化。

## 提交仓库区域

| 元素 | 说明 |
| --- | --- |
| 分支标签 | 显示当前分支，例如 `main`。 |
| 最近提交摘要 | 显示当前 HEAD 的提交说明，例如 `workspace: snapshot ...`。 |
| 更改数量 | 显示当前工作区有多少项更改。 |
| 提交消息输入框 | 输入本次提交说明。占位提示通常是 `消息(Ctrl+Enter 在 "main" 提交)`。 |
| `提交` 按钮 | 把当前更改提交到小说项目的本地 Git 仓库。 |

快捷键：

| 快捷键 | 作用 |
| --- | --- |
| `Ctrl+Enter` | 在当前分支提交。 |

## Agent 自动提交规则

Storydex 的设计目标是：每一次 Agent 执行结束后，如果检查到小说项目文件被修改，就自动创建本地 Git commit。

规则如下：

| 情况 | 结果 |
| --- | --- |
| Agent 修改了小说项目文件 | 执行结束后自动创建本地提交。 |
| Agent 没有修改项目文件 | 不创建空提交。 |
| 用户手动改了文件 | 可以在版本控制里手动提交。 |
| Git 不可用或仓库未初始化 | 无法提交，需要先初始化或修复 Git 环境。 |

桌面封装版会内置 MinGit，普通用户不需要额外安装 Git。

## 更改列表

`更改` 区域列出当前未提交的文件。

| 状态 | 含义 |
| --- | --- |
| `M` | Modified，已有文件被修改。 |
| `U` | Untracked/Added，新文件或未跟踪文件。 |
| `D` | Deleted，文件被删除。 |

文件行通常显示：

| 内容 | 说明 |
| --- | --- |
| 文件名 | 被修改的文件名。 |
| 相对目录 | 文件所在位置。 |
| 状态字母 | 本次变化类型。 |

点击更改文件行，会在中间文件区域打开 Git review / Diff 视图。

## Diff 预览

Diff 用于查看文件哪里变了。

| 操作 | 结果 |
| --- | --- |
| 点击 `更改` 里的文件 | 打开该文件的 Diff。 |
| 展开文件 | 查看具体修改片段。 |
| 收起文件 | 隐藏该文件修改详情。 |

Storydex 的 Diff 预览应优先展示修改区域，而不是把完整文件全文都塞出来。这样更适合快速确认 Agent 改了什么。

## 图表区域

`图表` 区域显示最近的本地提交历史。

| 内容 | 说明 |
| --- | --- |
| 提交线 | 展示本地提交的先后关系。 |
| 提交说明 | 例如 `workspace: snapshot ...`、`story: update chapters...`。 |
| 作者/时间 | 提交者和提交时间。 |
| 短哈希 | 例如 `a657fa9`，用于识别某次提交。 |

点击旧提交时，Storydex 会要求确认是否回退到该版本。回退前系统会先保留当前状态的本地备份分支，再恢复到所选提交。

## 常用操作

### 初始化版本控制

1. 打开小说项目。
2. 进入 `版本控制`。
3. 点击 `初始化本地仓库`。
4. 等待初始化完成。
5. 确认出现分支名和提交区域。

### 手动提交当前修改

1. 进入 `版本控制`。
2. 查看 `更改` 列表，确认修改文件符合预期。
3. 在提交消息输入框写说明，例如 `story: update chapter 3 draft`。
4. 点击 `提交`，或按 `Ctrl+Enter`。
5. 提交完成后，更改列表应变为空或减少。

### 查看某个文件的 Diff

1. 展开 `更改`。
2. 点击目标文件。
3. 在中间区域查看修改片段。
4. 如果内容很多，只看高亮的修改区域即可。

### 回退到旧版本

1. 展开 `图表`。
2. 找到想恢复的旧提交。
3. 点击该提交。
4. 阅读确认提示。
5. 确认后，Storydex 会先创建备份分支，再恢复到所选提交。

回退是高影响操作，建议只在确认当前内容需要撤回时使用。

## 和 Agent 执行浮条的关系

Agent 执行过程中，如果本轮有文件修改，浮条的“已修改内容”应显示本轮修改文件。点击文件后，可以在中间区域查看本轮 Diff。

版本控制则是更完整的版本视图，用于查看当前工作区和历史提交。

## 注意事项

- 版本控制不会自动上传到远程仓库。
- 如果用户把小说项目复制到其他电脑，`.git/` 目录也需要一起复制，历史记录才会保留。
- 不要把 Storydex 源码仓库的提交规则和小说项目提交规则混在一起。
- Agent 自动提交只应针对当前小说项目文件，不应提交 Storydex 源码文件。
- 如果自动提交失败，优先检查 Git 是否可用、项目是否初始化、本轮是否真的有项目文件变化。

## 可以这样问 Agent

```text
请总结本轮修改了哪些小说项目文件，并说明每个文件变化的目的。
```

```text
请查看当前版本控制，告诉我最近 5 次提交分别做了什么。
```

```text
请帮我检查当前更改列表，判断是否适合提交。
```

## 常见问题

### 为什么明明生成了内容，但更改列表为空？

可能原因：

1. Agent 还没有执行结束。
2. 文件写到了当前小说项目目录之外。
3. Git 状态还没刷新。
4. 项目没有初始化本地 Git。
5. 写入失败或被权限/路径问题阻止。

先点击 `刷新`，再检查资源浏览器里的文件是否真的变化。

### 为什么自动提交没有出现？

自动提交只在 Agent 执行结束后检查。如果执行还在进行中，或本轮没有项目文件变化，就不会提交。

### 可以推送到远程吗？

当前说明中的版本记录是纯本地用途。Storydex 不会默认把用户小说项目推送到远程仓库。

### 初始化 Git 会影响系统 Git 吗？

不会。桌面版会使用内置 MinGit，主要作用于当前小说项目目录。
